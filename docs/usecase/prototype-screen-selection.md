# ユースケース: プロトタイプの生成・更新（画面選択 / 微調整 / 画面追加）

プロトタイプ（プレビュー）の生成・更新には次の 3 ユースケースがある。
対象画面は `/studio/[id]/prototype`（AWSエンジン、生成は非同期ジョブ）。

| UC | 名称 | 起点 |
| --- | --- | --- |
| UC-生成1 | 初期生成（画面を選択して生成） | プレビュー未作成 |
| UC-更新1 | 既存画面の微調整 | プレビューあり |
| UC-更新2 | 未生成だった画面の追加（追記型） | プレビューあり・一部画面が未生成 |

## 共通の事前条件

- ユーザーがログイン済みで、対象プロジェクトの所有者である。
- 分析パイプラインでナビゲーション（メインナビ＝画面一覧）が生成済みである。
- モデル設定（基準／工程別）が読み込まれている。

---

## UC-生成1. 初期生成（画面を選択して生成）

| 項目 | 内容 |
| --- | --- |
| アクター | ビルダー利用者 |
| 目的 | 作りたい画面だけを選んでプロトタイプを生成する。出力量を抑えて途中切れ／タイムアウトを避ける。 |
| 選択単位 | **画面（ナビゲーション項目）のみ**（ユースケース単位の選択は行わない） |

### 基本フロー

1. プロトタイプ画面を開く。
2. システムはナビゲーション項目を**全画面選択済み**で「生成する画面」チップ群に表示する（`選択数/総数`）。
3. ユーザーが作りたい画面に絞り込む（`全選択` / `全解除`）。
4. 「プレビューを生成」を押す。
5. システムは生成ジョブ（`kind="prototype"`, `mode="create"`）を起動し、`jobId` を即返す。
6. 生成本体は `after()` でナビを**選択画面に絞った**コンテキスト＋「指定画面だけを過不足なく実装」指示でストリーミング生成する。
7. 進捗（受信文字数、`@screen` マーカー由来の「生成できた画面」）をライブ表示する。
8. 完了で HTML を保存し、プレビュー（`srcDoc`）と「生成された画面」一覧を表示する。

### 代替・例外

- **A1. 全画面選択（既定）**: 絞らなければ全画面を生成（`navigation` を絞らず `selectedScreens` も送らない）。
- **A2. チャット再生成**: 統合チャットからの再生成（`override.nav`）は選択フィルタを適用しない。
- **A3. 生成中の離脱**: 生成はサーバ側で継続。再訪時に進行中ジョブを復帰。
- **E1. 未選択**: ナビがあるのに 0 件選択だと「プレビューを生成」を無効化（「1つ以上選択してください」）。
- **E2. 途中切れ**: 出力上限（約16kトークン）到達で `finishReason="length"` を検知し警告表示。
- **E3. タイムアウト**: 関数上限（Hobby 300秒）超過で打ち切り→「画面を絞って分割」。

## UC-更新1. 既存画面の微調整

| 項目 | 内容 |
| --- | --- |
| アクター | ビルダー利用者 |
| 目的 | 生成済みプレビューに修正指示を出し、見た目・挙動を微調整する。 |
| 前提 | プレビュー（`prototypes.html`）が生成済み。 |

### 基本フロー

1. ユーザーが修正指示を入力する（例: 「サイドバーを青く」）。
2. 「更新」を押す。
3. システムは更新ジョブ（`kind="prototype"`, `mode="update"`）を起動し、**現在のHTML＋指示**から構成・データ・挙動を保ったまま**完全なHTML全文**を作り直す（差分ではない）。
4. 完了で HTML を保存し、プレビューを更新する。

### 代替・例外

- **A1. 反復**: 何度でも指示を重ねて微調整できる。
- **E1. 途中切れ／タイムアウト**: 出力は全文再出力のため、規模が大きいと UC-生成1 の E2/E3 と同じ上限に当たる。

## UC-更新2. 未生成だった画面の追加（追記型）

| 項目 | 内容 |
| --- | --- |
| アクター | ビルダー利用者 |
| 目的 | 初期生成で作らなかった画面を、**既存プレビューを保ったまま**後から追加する。 |
| 前提 | プレビューはあるが、目的の画面が未生成（初期生成で未選択 等）。 |
| 方式 | **(b) 追記型**: 既存HTMLを維持し、不足画面だけを足す（全体の作り直し＝(a)再生成型は採らない。それは UC-生成1 に吸収）。 |

### 基本フロー

1. ユーザーが追加したい画面（未生成の画面）を指定する。
2. システムは更新系の生成（`mode="update"` 相当）を、「**既存の構成・画面・データを維持し、指定した画面を追加する**」指示で起動する。
3. 生成本体は **現在のHTML＋追加指示**から、既存画面を保持しつつ新画面のビューとナビ項目を加えた**完全なHTML全文**を出力する。
4. 完了で HTML を保存し、追加された画面を含むプレビュー／「生成された画面」一覧を表示する。

### ビジネスルール

- **BR1. 継続性優先**: 既存画面・既存の手修正を保持する（全文再出力だが「維持」を指示）。これが (a) 再生成型ではなく (b) を採る理由。
- **BR2. コスト特性**: 入力に既存HTML全体を読むためトークンは多めになる。出力は全文のため、最終画面数が多いと UC-生成1 と同じ 16k/300s 上限に当たる。→ **追加したい画面＋最小限**に絞って実行する。
- **BR3. 範囲**: 1 回の追記で複数画面を足してもよいが、上限に注意（大量なら分割）。

### 実装メモ

- 現状の更新フロー（`streamUpdatePrototypeHtml`, `mode="update"`）を「画面追加」意図の指示で流用できる。**専用の導線（未生成画面の選択→追加）UI は今後追加**。

---

## ビジネスルール（横断）

- BR-共通1. 生成・更新・追記はいずれも**非同期ジョブ**で実行し、画面遷移・リロードでも止まらない。
- BR-共通2. プロトタイプは**単一の自己完結HTML**。更新・追記は差分ではなく**全文を再出力**する（既存維持はプロンプト指示で担保）。
- BR-共通3. 出力上限は約16kトークン（Hobby 300秒に収まる量）。超過は `finishReason="length"` で検知し、画面を絞った分割を促す。
